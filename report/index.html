Where it needed a human: (1) picking the 100 apps and their categories came from the assignment brief, not the agent; (2) grading the 22-app verification sample against real docs was done by hand — the agent never grades its own homework; (3) apps where the docs site blocked automated fetching (bot walls, JS-gated content, login-required docs) were flagged low-confidence and, where the browser re-check still couldn't resolve it, the "notes" field says so explicitly instead of guessing silently; (4) genuinely ambiguous "self-serve vs gated" calls (e.g. free tier exists but the useful endpoints are plan-gated) were spot-checked manually.

Proof — this is a runnable pipeline, not a one-off script

4Verification: did accuracy actually improve?

Human-graded sample of 22 apps, authentication, self_serve, buildable checked per app (66 field checks total).

67%
first-pass accuracy (before verify.py)
65%
final accuracy (after browser-use re-check loop)

Read honestly: on this sample the two numbers are within one field-check of each other (a single flip on 66 checks) — the browser-use loop did not measurably move the headline number. We're reporting that flat, not smoothing it over. The per-field breakdown below shows why: authentication is where nearly every miss lives, and on inspection most of those are the agent naming the right auth *family* with the wrong specific term (e.g. it says "Token", the docs say "Bearer Token" or "Personal Access Token") rather than being substantively wrong — an artifact of exact-string scoring, not a comprehension failure. Separately, there's a real, repeated pattern worth flagging: the agent over-guesses self_serve_trial when the docs actually say self_serve_free, several times across unrelated apps.

6Limitations & honesty

What this agent got wrong or couldn't resolve

See the miss list in section 4. Beyond that: a handful of docs sites in the set (bot-walled or requiring login, e.g. some finance/data-partner apps) returned little to no fetchable text, so those records lean on the model's prior knowledge and are flagged with lower confidence rather than presented as verified fact.

Intentionally not built (see README for why)
  • No parallel/worker-pool execution — sequential is fine at 100 apps and keeps failures easy to read.
  • No multi-source consensus beyond Composio registry + docs fetch + browser re-check.
  • No scheduled re-runs / continuous monitoring — this is a point-in-time research snapshot.
